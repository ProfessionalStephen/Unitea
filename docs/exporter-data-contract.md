# Exporter Data Contract

This dashboard has two separate data paths:

- Vercel nightly snapshots: generated by `/api/cron/send-briefings` from live Pipedrive, stored in Vercel Blob under `snapshots/YYYY-MM-DD.json`, and read by `/api/snapshots/series` for the KPI Trend block.
- Ops insights export: generated outside this app into `src/data/ops-insights.ts`, used for outcomes, cancellations, blockers, cycle times, and range cards.

## Vercel Nightly Snapshot Fields

The nightly snapshot should keep these fields current every weekday cron run:

- `date`: ET business date, `YYYY-MM-DD`.
- `capturedAt`: ISO timestamp of the Vercel run.
- `dataSource`: `live` or `simulated`; production trend needs `live`.
- `totalActiveJobs`: open active deals only, excluding terminal/completed stages and ignored boards.
- `totalPipelineValue`: value of current active/open pipeline deals.
- `endToEndDays`: sum of active stage average days, excluding terminal/completed stages and ignored boards.
- `wonThisWeek`, `wonThisWeekValue`, `wonLast30d`.
- `lostLast30d`, `lostLast30dValue`, `cancellationRate30d`.
- `activitiesDueToday`, `activitiesOverdue`, `callsDueToday`.
- `installsCompletedYesterday`, `installsScheduledThisWeek`.
- `permitsSubmittedThisWeek`, `sentToPermittingToday`.
- `nmaSubmittedThisWeek`, `serviceRequestsToday`, `techniciansScheduledToday`, `inspectionsScheduledToday`.
- `boardData[boardName]`: compact per-board data for region filtering and trend recalculation.
- `boardData[boardName].totalDeals`: active deals in that board.
- `boardData[boardName].totalValue`: active value in that board.
- `boardData[boardName].avgDays`: active weighted board average.
- `boardData[boardName].stages[]`: `name`, `count`, `avgDays`, `totalValue`, `stuckCount`; no per-deal arrays in snapshots.
- `pipelines`: live pipeline id/name list after ignored boards are removed.

## Ops Insights Export Fields Needed

The external exporter should include these aggregate fields so homepage/report ranges work without guessing:

- `generated`, `dataFreshThrough`, `source`, and an `exportRules` object with rule versions.
- `funnel.outcomes` for all-time lifecycle outcome counts.
- `windows[days].outcomes` for `30`, `60`, `90`, `180`, `365`, and optionally `730`/all-time if different from all-time funnel.
- `reporting.windows[days].winRate`, `completed`, `cancelled`.
- `reporting.windows[days].cancelRate`.
- `reporting.windows[days].inspectionFailRate`, `inspectionEvents`, and preferably `inspectionRedTags` count.
- `windows[days].redFlags.total`, `records`, and `categories` for blockers/setbacks.
- `windows[days].cycleTimes[]` with `key`, `label`, `median`, `mean`, `n`.
- `cancellations.monthly[]` with `month`, `count`, and `reasons[]`.
- `cancellations.monthly[].reasons[]`: `reason`, `count`, `pctOfMonth`, `pctOfSelectedRange` if exporter can calculate it.
- `cancellations.where[]`: `board`, `cancels`, `pct`, `medianDays`.
- Optional but recommended: `regions.FL` and `regions.CA` mirrors for all reporting and ops windows, so Florida-only views are truly region-specific instead of only board-filtered live metrics.

## Dirty Data Cleaning Rules To Version

These rules should not live only in someone's memory or in an unpublished exporter script. Include them in `exportRules` and keep them versioned:

- Ignored boards: `System Monitoring` is old/duplicate data and must be excluded from UI and metrics.
- Terminal boards: `Work Completed Not US customer` is not active work.
- Terminal stages: completed, PTO-paid, invoice-sent, processed cancellation, and serviced stages must be excluded from active work.
- Region mapping: board region comes from `shared/domain/boards.ts`; currently only `California` is CA and Florida is the default HQ view.
- Red-Tag definition: `Inspection Red-Tags` are municipality-issued inspection/code failures only, tied to the Inspection board/stage data.
- Permitting setbacks definition: permit declines, missing documentation, plan/application corrections, and permitting compliance issues are not Red-Tags.
- Deal status filters: active pipeline metrics use open deals only; win/loss rates use won/lost events in the selected date window.
- Date basis: snapshots and dashboard windows should use Eastern Time dates.
- Deduplication: define the key used to dedupe notes/events/deals, such as Pipedrive deal id plus event/note id.
- Test/demo data: define known test records, deleted deals, duplicate imports, and legacy board paths that should be excluded.
- Missing dates: define whether to drop records with no reliable event date or bucket them as unknown; do not silently mix them into dated windows.
- Outlier handling: define whether extreme ages are capped, excluded, or reported separately. Do not cap active pipeline stage age without labeling it.

## Why KPI Trend May Not Update

The current app reads KPI Trend from `/api/snapshots/series`, not directly from Pipedrive. The block will not move unless that endpoint returns at least two stored snapshots inside the selected range.

Common causes:

- No Vercel Blob snapshots exist yet, or only one exists.
- The cron has not run since deployment, or the Vercel cron did not fire at its daily `0 11 * * *` UTC schedule.
- `CRON_SECRET`, `PIPEDRIVE_API_KEY`, `PIPEDRIVE_DOMAIN`, or Vercel Blob env vars are missing in Vercel.
- The cron fires but Pipedrive pull fails, causing no `live` snapshot write.
- The snapshot write fails, visible in Vercel function logs as `Snapshot write failed`.
- The user is not signed in, so `/api/snapshots/series` is blocked by `requireSession`.
- The selected range is too narrow for the stored snapshot dates.
- The currently deployed version may not include the latest `region` support until this branch is deployed.

Recommended quick checks in production:

- Open `/api/snapshots/list` while signed in and confirm `count >= 2`.
- Open `/api/snapshots/series?days=90&region=FL` while signed in and confirm `status: "ok"` and at least two rows.
- Check Vercel logs for `/api/cron/send-briefings` and confirm the response includes a `snapshot` object, not `snapshot.error`.
